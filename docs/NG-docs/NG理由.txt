----------------------------------------------------------------
■ NGの核心
----------------------------------------------------------------
配信の起動を GitHub Actions の schedule（cron）に依存していたが、
指定時刻どおりに実行されなかった。

GitHub Actions の schedule は、特に「無料の public リポジトリ」では
実行時刻が保証されない仕様であり、数時間規模の遅延・発火スキップが
公式に起こり得る。cron の記述自体は正しくても punctual に動かせない。


----------------------------------------------------------------
■ 実測データ（新スケジュール 07:37 / 12:07 / 17:53 JST 設定後）
----------------------------------------------------------------
予定(JST)     実際に走った時刻(JST)              ズレ
07:37         ほぼ発火せず                        —
12:07         ほぼ発火せず                        —
17:53         07-04 19:35 / 07-03 20:53           +1.5〜3時間
（その他）    07-04 17:14, 07-04 00:58 など        不規則

→ 朝・昼がほとんど発火せず、夜〜深夜に不規則に固まって実行された。
→ cron の「分」をずらしても（07:00→07:37 等）改善しなかった。

■ 今後の対応方針（新設計）
----------------------------------------------------------------
起動トリガーを「外部スケジューラ → GitHub workflow_dispatch」に変更する（A案）。

  cron-job.org（クラウド・分単位で正確）
      └─ POST → GitHub API (workflow_dispatch)
              └─ GitHub Actions が news_bot.py を実行 → LINE配信

- GitHub 側の schedule: は廃止（残すと遅延した定期実行が二重配信の原因になるため）。
- workflow_dispatch（API/手動）起動は schedule のような大幅遅延の対象にならず、
  通常は1分以内に開始するため、時刻の正確さは cron-job.org が担保する。
- cron-job.org に持たせる GitHub トークンは、対象リポの Actions 書き込みだけに
  絞った Fine-grained token を使用し、漏洩時の被害を最小化する。
- ローカルPCは日々の配信に不要（全処理はクラウド側で完結）。


----------------------------------------------------------------
■ 未反映の既知課題（別途対応予定）
----------------------------------------------------------------
重複配信対策が「将来の拡張案」止まり。
HOURS_WINDOW=7時間 と配信間隔（4.5h / 5.8h / 13.7h）の関係上、
- 朝と昼で取得ウィンドウが重なり同一記事が重複配信され得る
- 夜間（17:53〜翌00:37 付近）は取りこぼしが発生し得る
- 重複除外(seen_titles)は1回の実行内でしか効かない（実行をまたげない）
→ 根本解決は「実行をまたいで配信済みを記録する（クロスラン重複排除）」。
